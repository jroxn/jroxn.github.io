两台电脑截图，导致图片丢失了一部分😭

起因：开发同学使用sqlline.py操作HBase报错

首先登陆MapReduce服务控制台查看报错：

参考的排错文档 并没有用。
按文档所说zookeeper，时间，hdfs等都是好的

根据报错信息查看服务器RegionServer是好的

从这里也能看到各项数据正常，但是RegionServer时好时坏

如下图所示（怀疑是拉起之后就被干掉，因为上图的HRegionServer也会消失）

这里进行了重启大法，大法GG😩

在mrs控制台找到hbase服务，进入监控页面，可以看到好几张表处于被打开的状态

往下可以看到有25张表处于RIT状态！！！

服务器日志也能看到OPENING状态的表：

就是这个鬼，参考文章实际并没有解决🤟
因为我的集群hbase是2.1.1版本，很多命令被2.0+的hbase抛弃了，文章中的命令我这里执行报错：不存在
另外一篇文章给了解决思路，查看日志是否是内存泄漏导致的RIT

这里的截图是错的，实际修改的是ThriftServer，原来的Xms128M Xmx128M修改为Xms1024M Xmx1024M

修改之后重启相关服务HBase Hue，修改完会提示重启，按照提示来就行
3分钟内，告警消失，HBase恢复

这里解决launcher.log过大350G+(磁盘空间告警)

这里解决熵告警： 

ps –ef | grep "haveged" | grep -v "grep"
ps -ef | grep "haveged"
yum -y install haveged
systemctl start haveged
systemctl enable haveged.service
cat /dev/random | od –x
cat /opt/mrs_clusterinfo 
cat /proc/sys/kernel/random/entropy_avail

解决磁盘满导致的数据不同步告警： 

cd /srv/BigData/journalnode/hacluster/current/
rm -rf edits_*
rm committed-txid 
cat /etc/hosts

scp root@192.168.1.73:/srv/BigData/journalnode/hacluster/current/edits_000* ./
